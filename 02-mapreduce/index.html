Map / Reduce
in RavenDB

How aggregation moves out of query time and into the index — and why that changes everything

Indexes Aggregation Multi-Map OutputReduceToCollection C# · RQL RavenDB 7.x
The Problem

Aggregation Is Expensive

Traditional approach

  • Scan a collection at query time
  • Group, count, sum — for every request
  • More data → slower queries
  • Popular items hit the hardest
  • Workaround: background jobs + caching + cache invalidation + delay…

The insight

What if aggregation happened incrementally, as documents change — not all at once when you query?

You'd pay a small cost per write, and then reads would be instant regardless of dataset size.

That's exactly what Map/Reduce indexes do.

Concept

What Is Map / Reduce?

📦

Analogy: Imagine a warehouse counting inventory. Instead of one person counting everything at the end of the day, every worker maps their section — "I have 5 red boxes" — and a supervisor reduces all reports into a total. Change one box? Only that section re-reports.

📄
MAP
Run a function over each document. Extract and emit intermediate key-value pairs.
→
⚙️
REDUCE
Group by key. Aggregate the intermediate results into a final value per group.
→
⚡
QUERY
Reads hit pre-computed results. Zero aggregation cost at query time.
Architecture

How RavenDB Does It Differently

Traditional MapReduce (Hadoop-style)

  • Distributes work across a cluster
  • Batch jobs — runs periodically
  • Results are stale until the next run
  • Designed for petabyte-scale ETL

RavenDB Map/Reduce

  • Runs on each node independently
  • Triggered incrementally on every write
  • Results are always fresh
  • Designed for real-time query serving

RavenDB uses MapReduce to break computation across time, not across machines.

Code Example

Basic Map/Reduce Index

// Count products per category
public class Products_ByCategory
    : AbstractIndexCreationTask<Product, Products_ByCategory.Result>
{
    public class Result { public string Category; public int Count; }

    public Products_ByCategory()
    {
        // MAP — emit one entry per document
        Map = products => from p in products
                          select new Result {
                              Category = LoadDocument<Category>(p.Category).Name,
                              Count    = 1
                          };

        // REDUCE — group and sum incrementally
        Reduce = results => from r in results
                            group r by r.Category into g
                            select new Result {
                                Category = g.Key,
                                Count    = g.Sum(x => x.Count)
                            };
    }
}
// Query the index — reads pre-computed results, no aggregation at query time
from index 'Products/ByCategory'
order by Count desc

// Filter to a specific category
from index 'Products/ByCategory'
where Category = "Electronics"

// Sample result document
{
    "Category": "Electronics",
    "Count": 142
}
Advanced Feature

Multi-Map / Reduce

Aggregate across multiple collections in a single index — unified, consistent results.

// Count Companies, Suppliers, and Employees per City
public class Cities_Details
    : AbstractMultiMapIndexCreationTask<Cities_Details.IndexEntry>
{
    public class IndexEntry {
        public string City; public int Companies, Employees, Suppliers;
    }
    public Cities_Details() {
        AddMap<Employee>(e  => e.Select(x => new { City = x.Address.City, Companies=0, Employees=1, Suppliers=0 }));
        AddMap<Company>(c   => c.Select(x => new { City = x.Address.City, Companies=1, Employees=0, Suppliers=0 }));
        AddMap<Supplier>(s  => s.Select(x => new { City = x.Address.City, Companies=0, Employees=0, Suppliers=1 }));

        Reduce = results => from r in results
                            group r by r.City into g
                            select new IndexEntry {
                                City      = g.Key,
                                Companies = g.Sum(x => x.Companies),
                                Employees = g.Sum(x => x.Employees),
                                Suppliers = g.Sum(x => x.Suppliers)
                            };
    }
}
// Query the unified view — all three collections reduced into one result per city
from index 'Cities/Details'
order by Companies desc

// Filter to a specific city
from index 'Cities/Details'
where City = "London"

// Sample result — one document per city, data drawn from three collections
{
    "City":      "London",
    "Companies": 28,
    "Employees": 94,
    "Suppliers": 15
}

// No JOINs. No cross-collection scans. One index, always fresh.
Advanced Feature

OutputReduceToCollection

Reduce results materialise as real documents in a named collection — enabling recursive roll-ups, subscriptions, and ETL on top of aggregated data.

// Write daily sales summaries as documents
public class DailySales
    : AbstractIndexCreationTask<Order, DailySales.Result>
{
    public DailySales() {
        Map = orders => from o in orders
                        select new {
                            Date    = o.OrderedAt.Date,
                            Revenue = o.Lines.Sum(l => l.PricePerUnit * l.Quantity),
                            Count   = 1
                        };
        Reduce = results => from r in results
                            group r by r.Date into g
                            select new {
                                Date    = g.Key,
                                Revenue = g.Sum(x => x.Revenue),
                                Count   = g.Sum(x => x.Count)
                            };

        // ✨ Output to a real collection
        OutputReduceToCollection = "DailySalesResults";
        PatternForOutputReduceToCollectionReferences
            = r => $"sales/daily/{r.Date:yyyy-MM-dd}";
    }
}

Output documents are queryable like any collection — or load by predictable ID. Chain another index on top for monthly or yearly roll-ups.

// Query the output collection directly
from DailySalesResults
where Date >= "2024-01-01"
order by Revenue desc
limit 30

// Load a single day by its predictable document ID
load "sales/daily/2024-01-15"

// Chain a monthly roll-up index on top of DailySalesResults
from index 'MonthlySalesRollup'
where Month = "2024-01"
order by Revenue desc

// Sample result document
{
    "Date":    "2024-01-15T00:00:00.0000000",
    "Revenue": 48250.00,
    "Count":   312
}

What you can do with output docs

  • Index them again → recursive roll-ups (daily → monthly → yearly)
  • Run data subscriptions on them (real-time consumers)
  • Apply ETL to push to external systems
  • Query them directly like normal documents

Real-World Use Cases

Where Map/Reduce Shines

📊 Dashboards & Analytics

Revenue by region, orders per day, customer rankings — always pre-computed, served in microseconds.

🏷️ Inventory & Catalog

Count products per category, track stock levels by warehouse — consistent across millions of SKUs.

📅 Time-Series Roll-ups

Daily → monthly → yearly summaries via chained OutputReduceToCollection indexes.

🏙️ Cross-Collection Stats

Multi-Map indexes: combine Employees, Companies, Suppliers into one unified view per city.

📦 Order Intelligence

Revenue per product, order frequency per customer — queries that would normally require heavy GROUP BY scans.

🔄 Event Sourcing

Maintain live read-model projections from event streams without a separate projections service.

Summary

Why RavenDB's Map/Reduce Is Unique

Capability Traditional DB RavenDB
Aggregation cost at query time High — full scan Zero — pre-computed
Result freshness Stale (cache/batch job) Always current (incremental)
Multi-collection aggregation Complex JOINs Multi-Map index — built in
Recursive roll-ups Separate jobs / ETL OutputReduceToCollection chains
Output as queryable documents Not natively Yes — full document operations
Infrastructure required Hadoop cluster / background workers Built into the database engine

RavenDB turns expensive aggregation into a first-class, zero-cost query primitive — no background jobs, no caches, no stale data.
